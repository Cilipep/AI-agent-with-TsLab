# FastPath: TSLab Agent Workflow

Use this file when context is tight or the skill loader failed.

This file is front-loaded. On the first read for a short localhost or exact-artifact task, stop after the first 120 lines unless one specific later keyword is still needed for the current live blocker. The deeper sections are reference-only; do not page through this file repeatedly before the next concrete localhost mutation or verification step.

If the active runtime or tool surface is Claude/openclaude, read `CLAUDE-FastPath.md` once for tool and shell quirks. Keep the common TSLab workflow in this file.
If you manually open follow-up AGENTS/skill files, stay inside the current workspace and do not assume parent/source paths exist outside it.
In packaged `ai-agent` workspaces, the canonical follow-up docs are the local relative copies such as `./AGENTS-ScriptEditing.md`, `./AGENTS-RunAndAnalyze.md`, and `./.opencode/skill/tslab-script-authoring/SKILL.md`.
Do not open `./.codex/skills/...`, user-home skill paths, or globally advertised legacy skills such as `$tslab-webapi` when the current packaged workspace already ships local `.agents/skills/tslab*` or `./.opencode/skill/tslab*` copies. Stay inside the packaged local skill set.
If you need one of those hidden local skill paths, open the exact known relative path directly. Do not use wildcard glob/search calls just to prove that `./.agents/...` or `./.opencode/...` exists; some toolchains skip hidden directories and produce false negatives.
If the active runtime is Claude/openclaude, the Claude-specific delta in `CLAUDE-FastPath.md` covers tool and shell quirks, including the ban on using `.claude/projects/...` or `tool-results/*.txt` as source-of-truth.

## Core rule

- Treat canaries as generic regression classes, not as prompt-specific targets. Do not optimize behavior for prompt ids or one canned wording. Pass by following the live API contract and producing the right artifact, not by pattern-matching one test prompt.

- First localhost JSON rule: do not use a generic `write`/editor tool for request-body files. In Windows PowerShell, write request bodies with the current shell under `./tmp/...` (for example `Set-Content -Encoding utf8 ./tmp/body.json ...`) and then post through `./api-json.ps1`, `./script-api.ps1`, or `./run-lifecycle.ps1`.
- Never attempt `/tmp/*.json`, `/workspace/tmp/*.json`, `C:\tmp\*.json`, repo-root temp files, or parent-workspace temp files. If such a path attempt fails, immediately redo the same body under `./tmp/...`; the external failed write is not progress.
- For blank create, prefer `./create-script.ps1 "<leafName>" ["<parentPath>"]`. Only create a body file when that helper is insufficient, and then create that body with the current shell under `./tmp/...`.
- Workspace-local paths stay workspace-local. Read user-provided prompt/spec inputs from the exact relative path the prompt names, and write request bodies or scratch JSON as `./tmp/...`; never rewrite relative inputs to `/name`, `/tmp/...`, `/workspace/tmp/...`, `C:\tmp\...`, a repo root, drive root, or a parent workspace.
- Create localhost request-body JSON with the current shell under `./tmp/...`, then post it through the packaged helper. Do not use a generic editor/write-file tool for those request bodies; such tools often resolve `/tmp/...` or another sandbox-global path outside the packaged workspace.
- If a local file operation is rejected as `external_directory`, absolute-rooted, or permission-denied, the next action is to retry the same read/write with the workspace-local `./...` path. If a generic read/write tool keeps converting `./name` or `./tmp/<folder>/...` to `/name`, `/tmp/...`, `/workspace/tmp/...`, or another workspace, use the current shell with the relative `./...` path and continue from that file, not from memory, broader discovery, old artifact names, or absolute temp paths.
- If the immediately previous localhost helper output contained `HELPER_SUMMARY`, `INCOMPLETE_TURN_DO_NOT_ANSWER`, `finalAnswerAllowed=false`, `sameTurnIncomplete=true`, or `lifecycle-prebuild-stop`, that output is the active next-step contract. Do not answer the user, do not reread docs as a substitute for progress, and do not call lifecycle while it says repair is required. Execute the named same-artifact helper step first.
- If a helper or live response says `repair-after-run`, `lifecycle-run-requires-repair`, `sameTurnIncomplete=true`, or `apply-next-ops-before-answer`, the next assistant action must be a concrete same-artifact mutation/proof step. Do not end the turn with prose such as `I will fix...`; write the repair body under `./tmp/...`, POST it to `/ops`, and rerun proof before any final report.
- If the user prompt contains an exact final-proof `intent-check` JSON body, lifecycle success and metrics are only provisional. Final answer is forbidden until the exact body has been posted to `POST /api/scripts/{name}/intent-check` on the same artifact and returns `requiresRepair=false`; if it returns repair ops or `requiredNextMutationRoute`, execute that repair path first.
- `one phase = one shell step`
- The packaged workspace normally already contains `./tmp`. Do not spend a shell step on `mkdir ./tmp`, `New-Item ./tmp`, or `mkdir ./tmp -ErrorAction ...` before writing a request body. If a directory really is missing, create it with the syntax for the shell you are actually using; do not mix PowerShell-only flags such as `-ErrorAction` into Bash/POSIX-style commands.
- If a PowerShell hashtable, quoting, or inline JSON attempt fails while preparing a localhost request body, do not switch to a generic file/write helper and do not write `/tmp/*.json`, `/workspace/tmp/*.json`, or `C:\tmp\*.json`. Retry in the same current workspace with a shell command that writes `./tmp/<name>.json`, for example `@{request=@{dataSourceName='ByBit';securityId='BTCUSDT';interval='5m'}} | ConvertTo-Json -Compress | Set-Content -Encoding utf8 ./tmp/instrument-source.json`, then post that file through `./api-json.ps1`, `./script-api.ps1`, or `curl.exe --data-binary`.
- On chained refinement turns for an existing scaffold, start from compact current-state proof: `authoring-quality?compact=true`, the exact prompt `intent-check?compact=true`, `explain?summary=true`, `ui`, or the exact latest blocker route. Avoid full `/graph` or full `/json` reads unless a compact response did not contain the exact current block id, pane key, parameter, or mapping needed for the next `/ops` patch.
- If the prompt explicitly asks to load, open, or show an exact script in desktop, first call `POST /api/scripts/{name}/load`. Exact-script read routes such as `lab-options`, `parameters`, `graph`, `explain`, `ui`, `messages`, `logs`, and `authoring-quality` may auto-open/refresh desktop as a server fallback, but they only close a load/open request when the response message explicitly says `opened in desktop` or `refreshed in desktop`.
- If the user prompt gives an exact script/artifact name, the first create call must use that name byte-for-byte. Do not shorten it, translate it, normalize underscores, invent a friendlier alias, or create a helper script under a different name.
- On an ordinary exact-name create-run in Windows PowerShell, the first create call should be `./create-script.ps1 "<leafName>" ["<parentPath>"]` or `./api-json.ps1 POST /api/script-manager/scripts <bodyFile>`. If exact folder placement matters, pass the leaf name and parent path separately instead of one foldered string. Do not spend that first create step on raw `Invoke-RestMethod POST /api/scripts` or other guessed create routes.
- For exact script-scoped routes on one known artifact, prefer `./script-api.ps1 <METHOD> "<scriptName>" <route> [bodyFile]` on PowerShell or `./script-api.sh ...` on Bash/POSIX. These helpers choose the correct ordinary-vs-`/by-name` route shape automatically for foldered names, so you do not have to hand-assemble `/api/scripts/<route>/by-name/...`.
- For foldered scripts or names with spaces, do not hand-build `/by-name/...` URLs and do not inspect graph JSON through fragile PowerShell projections such as `$r.`. Use helpers such as `./script-api.ps1 GET @last-created explain?summary=true`, `ui`, `graph`, `authoring-quality?compact=true`, or the exact lifecycle route.
- If blank-create or an exact-name lookup returns a foldered `resolvedArtifactRouteName`, pin that exact full server-side path immediately. Do not switch back to a leaf-only `/api/scripts/{name}/...` route after create.
- A blank-create response is never a finished user-visible result. If that same response still says `BlankGraph`, `mustMutateCurrentArtifact=true`, `mustNotStopBeforeMutation=true`, or exposes `requiredNextMutationRoute`, that same turn must continue on the same artifact. Do not answer with only `script created`, `artifact ready`, or another blank-create status line.
- If the prompt asks for the first meaningful implementation turn or first working pass on a new artifact, blank-create or instrument-source alone is still not meaningful progress. That turn must already include the first live `POST /api/scripts/{name}/ops` scaffold on the same artifact. On strategy or trading prompts, that first meaningful scaffold must already include one real entry family, `RelativeCommission`, and one protective or profit-taking exit instead of only `Close`/`Open`/`High`/`Low`/Bollinger/helper indicators.
- If a fresh create prompt asks to run/backtest/report metrics, treat it as needing a runnable live trade path unless it explicitly says analysis-only, visual-only, indicator-only, dashboard, or asks to add/show an indicator on a chart without entry/exit/trading logic. A custom indicator overlay/control-pane task remains non-trading even if it asks for runnable proof or metrics. Preserve every prompt-required indicator/control block by wiring it into the visible chart or the trade path; do not leave it as a detached proof block and do not remove it just to silence cleanup.
- If the prompt asks to implement/build/upload a C# or DLL custom indicator, an existing handler/DLL with similar names is discovery only. Build and upload a fresh DLL from this current workspace before using those handlers, unless the user explicitly asked to reuse an existing DLL.
- On complex strategy prompts, the first live tradable scaffold should be one minimal working leg plus `RelativeCommission`, one protective or profit-taking exit, and required overlays. Do not build a full symmetric long/short system, adaptive refinements, or multiple parallel entry/exit branches before the first `validate -> build -> load -> run` proof succeeds on that simpler leg.
- If the prompt says to use offline, cached, local, or backtesting data for a provider, treat that as a mapping/runtime-data selection only. Do not read datasource `settings` or `schedule`, do not call unfiltered `GET /api/datasources` to choose another source, do not connect, disconnect, reload, reschedule, wait-loop, switch providers, or add `X-TSLab-Provider-Admin-Confirm`; use the cached data already available to the host. If datasource readiness must be checked, use at most one `GET /api/datasources/{name}/status` for the already-mapped/requested datasource, then return to same-artifact repair. `CompletedNoData`, disconnected status, `records=[]`, `isTradable=false`, or `instrumentsLoaded=0` is not proof by itself that cached history is missing; repair the current artifact's graph/source wiring/lab-options and rerun before reporting provider cache readiness as the blocker.
- Before that first live trade scaffold exists, do not burn the starter lookup budget on `Close`, `Open`, `High`, `Low`, `Highest`, `Lowest`, `StDev`, `EMA`, `Bollinger*`, or generic handler/catalog archaeology. Use the prompt-required entry family, `RelativeCommission`, and the protective/profit-exit family, or mutate immediately when those are already clear enough.
- If `requiredNextMutationRoute` already points to `POST /api/scripts/{name}/repair/authoring-quality`, that wrapper itself is the next step and needs no handcrafted repair file. Do not pause to create a manual `/ops` body or a copied `suggestedRepairOps[]` payload first.
- On prompts that ask for the first meaningful implementation turn or first working pass, do not stop at a starter scaffold alone. If the current artifact is now runnable, continue with `GET /api/scripts/{name}/authoring-quality` and then `validate -> build -> load -> run -> metrics-summary` before the final summary. If it is still not runnable, stop only after one exact live blocker from `authoring-quality`, `validate`, `build`, `load`, or `run` on that same artifact.
- For breakout, limit, or stop scaffolds whose prompt does not explicitly require a condition-wrapper family, prefer direct price-based non-market entry and exit blocks first. If you do choose a condition-driven family such as `OpenPositionIf*` or `ChangePositionIf*`, wire the required `Eq` gate in that same first batch. A first scaffold that immediately returns `OpenConditionNotConnected`, `BlocksMissingRequiredInputs`, `Entry1 input not connected`, `BlocksMissingSecuritySourceInput`, or another warning on the blocks you just added is still unfinished and must be repaired on that same artifact before stopping.
- The packaged `./create-script.ps1` and `./create-script.sh` helpers now persist the canonical created or reused artifact into `./tmp/last-script-artifact.json`. In the same workspace, immediate follow-up `./script-api.ps1 @last-created <route> ...`, `./script-api.sh @last-created ...`, `./run-lifecycle.ps1 @last-created`, or `./run-lifecycle.sh @last-created` keeps the helper pinned to that exact artifact without retyping the name from memory.
- The packaged `./create-script.ps1` and `./script-api.ps1` helpers also emit short `HELPER_SUMMARY` lines on stderr for `blank-create`, `instrument-source`, `authoring-quality`, `repair/authoring-quality`, and unfinished scaffold `/ops` responses. Treat those summaries as compact control signals for the next localhost step on the same artifact; do not ignore them and stop on `created`, `source mapped`, `starter scaffold added`, or a long AQ JSON dump.
- If a current localhost response also exposes machine-readable fields such as `readBudgetBeforeMutation`, `primaryMutationObjectives[]`, or `preferredDiscoveryRoutes[]`, follow those fields literally on the same artifact before more discovery. On starter-scaffold phases this typically means a small exact contract budget, `template-blocks` first, then immediate `/ops`.
- For LLM-facing contract reads, keep discovery responses compact: append `compact=true` to focused template/handler reads, for example `GET /api/template-blocks?query=OpenPosition&top=20&compact=true`, `GET /api/template-blocks/OpenPositionAtLimitItem?compact=true`, or `GET /api/handlers/High?compact=true`. Do not pipe full contracts through `ConvertTo-Json -Depth 10`; the compact response already contains `inputs[]`, `parameters[]`, and `suggestedAddBlockOp`.
- `Open` / `High` / `Low` / `Close` / `Volume` are OHLCV handler-backed source series. Use `GET /api/handlers/High?compact=true` or the `ConverterItem` + `typeName` shape returned by `/api/template-blocks?query=High&compact=true`; connect `Src` to `SECURITYSource` and do not keep searching template-block variants.
- Before any handcrafted `POST /api/scripts/{name}/ops`, keep the low-level contract fixed: canonical op-root fields are `blockId`, `typeName`, `blockType`, `category`, `position`, `fromBlockId`, `fromPort`, `toBlockId`, `toPort`, and `toInputName`. Do not send alias keys such as `from`, `to`, `block`, `key`, `fromBlock`, `toBlock`, `handlerType`, `handlerTypeName`, or `toPortNum` as replacements for those fields. If a live response gives `suggestedRepairOps[]`, `suggestedAddBlockOp`, `canonicalAddBlockOp`, or `suggestedSetOptimizationRangeOp`, post that shape instead of translating it from memory.
- `SetOptimizationRange` is not a `SetParam` variant. Use `paramInvariantName` from `/parameters`, `/block-parameters`, or a current `suggestedSetOptimizationRangeOp`; use `paramName` only with `SetParam`. New range ops should use the API fields `paramInvariantName`, `optimDataType`, optional `value`, `min`, `max`, `step`, and `usedInOptimization`; do not invent `parameterCodeName`, `optimMinValue`, `optimMaxValue`, or `optimStep` request fields.
- For handler or template-backed blocks, do not hand-author a generic `ConverterItem` plus `handlerTypeName` and hope it binds. Reuse the live `suggestedAddBlockOp` / `canonicalAddBlockOp` for that exact family, then wire it with canonical `Connect` / `ConnectByInputName`.
- For formula carriers such as `BoolCustomHandlerItem` and `DoubleCustomHandlerItem`, connect real upstream blocks and write `Expression` against stable upstream `blockId` / `codeName` tokens by default. `Input0`, `Input1`, and similar generic slots are metadata labels, not safe default runtime aliases, unless the same host/artifact has already proved that exact executable expression syntax.
- Position sizing is a graph contract, not a caption. `Shares` must receive a real numeric stream or host-proved constant/value block connected to the actual `OpenPosition*` / `ChangePosition*` `Shares` input; a `SetParam`, unconnected formula block, or formula wired only to `Input0`/`Input1` does not satisfy position size. Verify the live graph through `/explain`, `authoring-quality`, lifecycle, and run metrics before optimization.
- For repeated proof/repair loops, prefer compact diagnostics first: `GET /api/scripts/{name}/authoring-quality?compact=true` and `POST /api/scripts/{name}/intent-check?compact=true` with the same body. Use the full response only when a compact issue, route, or ops preview is insufficient; never dump full diagnostic JSON into the model context just to see `requiresRepair`, missing families, or the next route.
- If `readBudgetBeforeMutation=1` on the current artifact, that budget is literal and optional: the default is to mutate immediately on the same artifact. Spend at most one exact template-block or exact block-family lookup only when one still-missing family is genuinely unclear. Do not turn that single-budget state into another `authoring-quality`, `repair/authoring-quality`, `explain`, docs, or broad search loop.
- On strategy or trading prompts in that `readBudgetBeforeMutation=1` state, if the prompt, local source notes, or `preferredMutationFamilies[]` already make the starter family clear enough, skip the lookup and send the first `/ops` batch now. If you do use the single allowed lookup, it must target a starter trade family such as `OpenPosition`, `ChangePosition`, `ClosePosition`, or `RelativeCommission`, or the exact entry/exit family named by the prompt. Do not spend that lookup on source or indicator families such as `Close`, `Open`, `High`, `Low`, `StDev`, `EMA`, `Highest`, `Lowest`, `Bollinger*`, `ConstGen`, `Multiply`, `Less*`, or `Greater*` before the first live trade scaffold exists.
- On chained or continuation turns in the same session, do not emit large `Goal` / `Constraints` / `Progress` / `Next Steps` recap blocks between localhost steps. Keep narrative to one or two short lines and spend the budget on the next mutation or proof route on the same artifact.
- Once helper summaries or live API hints already point to `validate -> build -> load -> run`, prefer one `./run-lifecycle.ps1 @last-created` / `./run-lifecycle.sh @last-created` call on that exact artifact. If you already started manually, continue immediately with `./script-api.ps1 @last-created <route>` / `./script-api.sh @last-created <route>`; do not switch to raw `curl.exe` with guessed lifecycle routes, and do not spend that proof turn on a recap block first.
- During lifecycle, `validate`, `build`, and `load` success are not stopping points. A `load` response always means the same proof is still open: call `run`, then read `metrics-summary` and any required intent proof before final answer.
- If `HELPER_SUMMARY authoring-quality ... state=BlankGraph|SourceOnlyGraph|DisconnectedGraph ... next=POST /api/scripts/{name}/repair/authoring-quality` appears, the next step is that wrapper immediately on the same artifact. Do not reread docs, examples, handlers, or template catalogs first.
- If `HELPER_SUMMARY ops-warning ... OpenConditionNotConnected`, `HELPER_SUMMARY ops-warning ... Entry1 input not connected`, `HELPER_SUMMARY ops-incomplete ... mustRepairBeforeLifecycle=true`, or an `/ops` response says `mustRepairBeforeLifecycle=true`, lifecycle is forbidden. The next command must be the advertised `repair/authoring-quality` or one focused `/ops` gate/input repair on the same artifact; do not call `validate`, `build`, `load`, `run`, `metrics-summary`, or optimization first.
- If `/ops`, `validate`, `build`, or helper output reports `MissingChangePositionShares`, lifecycle and optimization are forbidden until the same artifact has a real numeric Shares input on that ChangePosition* leg. For fixed or optimization-ready add size, use a host-proved numeric source such as `ConstGen` with `Value`, wire the live `Src` / `SECURITY` stream into the constant if its contract requires input, then connect that output to `ChangePosition*.Shares` with `ConnectByInputName` `toInputName="Shares"` or `toPort=2`. For derived add size, connect real upstream numeric/position-state blocks into `DoubleCustomHandlerItem` and connect the formula output to Shares. Do not assume `SetParam` on `ChangePosition*` or an unconnected formula/value placeholder satisfied the Shares port.
- If `requiredNextMutationRoute` or `HELPER_SUMMARY` already points to `POST /api/scripts/{name}/repair/authoring-quality`, call that wrapper directly before any manual `/ops` body work. That wrapper itself needs no handcrafted repair file. Do not stop to write `./tmp/repair.json`, `/tmp/repair.json`, or a copied `suggestedRepairOps[]` payload before the wrapper call.
- If `HELPER_SUMMARY repair-authoring-quality ... starterScaffoldOnly=true` appears on a strategy prompt, the generic starter repair is done. The next step must be one manual `POST /api/scripts/{name}/ops` batch that adds a real entry family, `RelativeCommission`, and one protective or profit-taking exit on that same artifact.
- If `HELPER_SUMMARY authoring-quality ... focus=AnalysisIndicatorRepair` or `HELPER_SUMMARY repair-authoring-quality ... focus=AnalysisIndicatorRepair` appears, the artifact is non-trading analysis/indicator output. Repair only the reported data-flow, visualization, or prompt-required controls on the same artifact; do not add trade blocks just to satisfy generic strategy scaffolding.
- If that same helper summary also says `do-not-repeat-repair-authoring-quality-while-hasTradeBlocks=false`, follow it literally. Do not loop the generic wrapper again before the first live trade scaffold exists; move to one manual `/ops` batch on the same artifact instead.
- If `HELPER_SUMMARY authoring-quality ... focus=StarterScaffold ... next=POST /api/scripts/{name}/ops` or `HELPER_SUMMARY repair-authoring-quality ... starterScaffoldOnly=true ... defer-indicator-formula-family-search...` appears, do not spend the next step on `Highest*`, `Lowest*`, `StDev`, `Max`, `Min`, expression-template, or other indicator/formula family discovery. Also do not detour into constant/value or arithmetic helper discovery such as `ConstGen`, `Value*SourceItem`, `Multiply`, `Less*`, or `Greater*` before the first live trade scaffold exists. The next batch must establish the live trade scaffold first: entry + `RelativeCommission` + protective/profit exit on the same artifact.
- If build/runtime messages say `ClosePositionByStopItem`, `ClosePositionByProfitItem`, or a close block needs an order price, do not keep reconnecting raw current `Close`/`Open` into `Prc`. Raw `Close`/`Open`/`High`/`Low`/`Highest`/`Lowest` is not a stop/profit level by itself. Create a real numeric exit-level stream such as `EntryPrice +/- fixed`, percent, ATR, or another prompt-appropriate risk/target formula, connect that derived level into `Prc`, then rerun proof.
- If a protective/profit exit is already present but lifecycle says its order price is missing, the next batch is not another starter repair loop. Keep the close block and repair its `Prc` with a derived price-level stream on the same artifact.
- If `HELPER_SUMMARY authoring-quality ... focus=FormulaRepair ... next=POST /api/scripts/{name}/ops` or `HELPER_SUMMARY repair-authoring-quality ... focus=FormulaRepair ... next=POST /api/scripts/{name}/ops` appears, stay on one formula-repair `/ops` batch using the current artifact's exact block ids and upstream code names. Do not widen into `Formula`, `Prev`, `PrevN`, or broad handler/template contract search first.
- If the prompt points to a workspace-local spec, prompt file, or other local input artifact, read that exact local file before blank-create or mutate. Do not substitute memory, an external Desktop copy, a guessed sibling file, or a previous script name when the prompt already named a workspace-local path.
- If the prompt names only a leaf filename but the current workspace already contains that exact file under `./tmp/inputs/` or another prompt-provided local path, open that local copy immediately and use it as the source of truth. Do not keep analyzing a remembered external/Desktop original after the workspace-local copy already exists.
- In later repair/proof turns on the same artifact, treat that local-spec read as a one-time prerequisite, not a loop. If the exact file was already read in this artifact workflow, do not reread it, do not widen into sibling-file discovery, and do not restart broad local-file analysis before the next localhost mutation or proof step.
- If the prompt already supplied an exact `intent-check` body and that same body still returns `requiresRepair=true`, `requiredNextMutationRoute`, or non-empty `suggestedRepairOps[]`, the stage is still open. Do not stop at a minimal scaffold or progress summary.
- Treat wording such as "for example", "e.g.", or "such as" as optional implementation hints unless the exact prompt or exact `intent-check` body requires that concrete family. If authoring-quality and the exact intent-check are already clean, do not mutate the graph just to add the optional example; optional polish can easily break a live artifact.
- Phrases like "at least one useful separate analysis pane", "use ... and at least one ...", or "must include one useful ..." are requirements, not optional polish. If `intent-check` reports `missingRequiredVisualFeatures` such as `separate-pane`, lifecycle and metrics are provisional until the same artifact is repaired and the exact `intent-check` returns `requiresRepair=false`.
- If that exact `intent-check` response already exposes non-empty `suggestedRepairOps[]`, prefer `POST /api/scripts/{name}/repair/intent-check` with that same request body before manually copying those ops into `/ops`.
- If that exact prompt-specific `intent-check` already shows every required trade, visual, handler, block, or compressed-interval family as present but `authoringQualityRequiresRepair=true` or `authoring-quality` still reports finishing blockers such as `NoOptimizationMappings`, `MissingGraphLinkSource`, or `DuplicateGraphColors`, the artifact is still unfinished. Follow `requiredNextMutationRoute` exactly, usually `POST /api/scripts/{name}/parameterize`, `POST /api/scripts/{name}/visualize`, `POST /api/scripts/{name}/repair/authoring-quality`, `POST /api/scripts/{name}/repair/intent-check`, or one more `/ops` repair on the same artifact, then rerun proof before the final summary.
- If the prompt says parameters should be optimization-ready, convenient for later optimization, or that a later phase will optimize the same artifact, treat `NoOptimizationMappings` as a blocking task-level gap even when `MissingControlPane` stays non-blocking.
- On an optimization follow-up for an already-created artifact, do not spend the turn on fresh family lookup, template archaeology, or broad doc rereads. If the artifact is blocked, repair that same artifact first; otherwise read `GET /api/scripts/{name}/parameters`, confirm or configure ranges, then `POST /api/scripts/{name}/optimization/start` and stay on the returned `jobId` through `best` and `apply`.
- If that optimization follow-up had to repair the current artifact first, the turn is still open once the script becomes runnable. Return to `GET /api/scripts/{name}/parameters`, start one fresh optimization job, and continue through `best` and `apply` in that same turn when the prompt explicitly asked to optimize, compare rows, or apply the best result.
- On an improve/test/compare strategy task, every improvement candidate you present as an outcome must have live proof from this turn: either a current-artifact optimization result, or a copied/current variant that passed `validate -> build -> load -> run` plus metrics. Do not finish with "could try", "можно попробовать", "not tested", or similar candidate language unless the same paragraph cites live endpoint evidence that the candidate cannot be tested now, such as no matching handler/template route.
- If the task asks to improve and verify a weak strategy, keep a small tested-candidate table: baseline, exact changed parameter/graph idea, lifecycle state, trade count, net profit, Profit Factor, drawdown, and accept/reject. Untested ideas belong only in a clearly blocked/out-of-scope note with endpoint evidence; they are not an improvement result.
- If the user only asked for a historical quality check but you voluntarily test variants, the same rule applies once you present "all variants tested", "optimization results", or a tested comparison table. Do not append a "what could be tried next" list unless those items are also tested or explicitly blocked by live endpoint evidence.
- When `intent-check`, `authoring-quality`, or another live repair response already gives `suggestedRepairOps[]`, prefer the advertised wrapper first, especially `POST /api/scripts/{name}/repair/intent-check` for prompt-specific semantic repair and `POST /api/scripts/{name}/repair/authoring-quality` for generic structural repair. If you still stay manual, post them as a JSON object shaped like `{ "ops": [ ... ] }` or through `./api-json.ps1`. Do not send a bare JSON array.
- When repairing missing trade `Prc` inputs on the current artifact, prefer existing numeric level blocks whose ids, code names, or visual labels already match that leg's semantics, for example entry/open vs exit/close and long/buy vs short/sell, before falling back to a generic `Close*` stream. Do not auto-wire every missing `Prc` to one blind price block when the same graph already exposes more specific entry or exit levels.
- After two identical tool schema or validation errors on the same call shape, stop repeating that loop. Switch to another tool family or the direct localhost API route. This does not apply to a server-advertised warmup/desktop readiness response that says to retry in 1-2 seconds; use the bounded manager-not-ready retry rule above, then stop only that retry loop and move to editor-state diagnosis.
- On a short exact-artifact task, if you already opened this file once in the current session, do not reread it end-to-end before the next `/ops`, `/visualize`, `/parameterize`, or lifecycle step. Search for one exact keyword only when a concrete contract is still unclear.
- If a local prompt or AGENT file already enumerates exact follow-up file paths, read those exact files directly in that declared order. Do not list sibling directories or scan nearby files just to rediscover the same paths.

## Universal guardrails

- TSLab scripts and portfolios are server-side `/api/*` entities, not local files.
- Server-side TSLab scripts are graph artifacts mutated through `/ops`, `/repair/*`, `/visualize`, `/parameterize`, `instrument-source`, and `lab-options`, not text/code documents. Do not hunt pseudo-content routes such as `/api/scripts/get_code`, `/api/scripts/get_content`, `/api/scripts/read`, or guessed code-export endpoints during normal script authoring.
- On a blank or source-only strategy artifact, do not pause to find a script-code/content endpoint before the first scaffold. Continue the same artifact through `instrument-source`, `suggestedRepairOps[]`, or the next real `/ops` batch.
- If the prompt already gave datasource, security, and timeframe, do not burn steps on guessed security-discovery routes such as `/api/securities` or `/api/datasources/{name}/instruments` unless the current host docs or live API already proved that route. Use the documented source-mapping and script mutation flow you already have.
- When one concrete security lookup is still needed, the canonical route is `GET /api/agent-manager/securities?dataSourceName=...&query=...&limit=20`. Do not keep guessing alternate datasource-specific instrument-list endpoints after that route is already available.
- Do not scan the repo to discover server-side scripts, portfolios, or agents.
- Do not read or create local `scripts/*.cs` files for server-side authoring. Stale repo artifacts are not the source of truth.
- If the user explicitly asked for a custom indicator in C#, build it in a temporary workspace and upload the DLL. Do not place indicator source files in repo `scripts/`.
- Use the current shell's native HTTP client for small JSON bodies. If a doc or API hint shows a PowerShell example, treat it as one shell variant and send the same route/body with `curl` or another native client on macOS/Linux.
- Do not wrap API calls in nested shell invocations such as `powershell -Command`, `bash -lc`, or `sh -lc` when the current shell can call the route directly.
- Use `Invoke-RestMethod` only when the current shell is actually PowerShell. If the shell says `Invoke-RestMethod` is unknown, stop retrying it and switch immediately to `curl.exe` or another native client for that shell.
- If the current shell is not PowerShell, do not use PowerShell-only commands such as `Copy-Item`, `ConvertFrom-Json`, `Select-Object`, or `Out-Null`; use the shell's own file/HTTP commands or the shell-matching packaged wrapper instead.
- On Windows PowerShell, do not start with POSIX-only discovery commands such as `grep`, `wc`, `head`, or `ls -R`. Use `Get-ChildItem`, `Measure-Object`, `Select-Object`, `rg`, or other PowerShell-native equivalents instead.
- On Windows PowerShell, for localhost REST calls use `Invoke-RestMethod`, the packaged wrappers, or `curl.exe`. Do not use bare `curl`, because it resolves to the PowerShell web alias and frequently breaks `-s`, `-H`, or `-d` examples.
- On Windows, do not infer the shell from the OS alone. If the current tool behaves like Bash/POSIX, stay in Bash-compatible syntax even on Windows: do not append PowerShell formatters like `Out-String`, do not rely on `ConvertTo-Json`, and do not wrap ordinary localhost REST calls in `powershell -Command` just to manufacture JSON.
- On Windows PowerShell, do not pass raw wildcard path arguments such as `rg AGENTS*.md` or `rg .\path\*.md` to `rg`; that commonly produces `os error 123` instead of opening files. Use `rg pattern .`, `rg --files | rg <name>`, or `Get-ChildItem -Filter` first, then open the exact file.
- In shell steps, do not invent pseudo-commands such as `write -filePath`, `read -filePath`, or `edit` unless the current shell wrapper explicitly provides them. Use native shell file-write commands or the actual dedicated tool instead.
- On Windows PowerShell, create JSON body files with native commands such as `@'...'@ | Set-Content -Encoding utf8 <path>` or `Set-Content -Encoding utf8 <path> <json>`, then `curl.exe --data-binary "@file.json"`. Do not assume a shell-level `write` helper exists.
- For localhost API JSON, use full structured reads (`Invoke-RestMethod`, `ConvertFrom-Json`, or full `curl.exe` JSON piped into a JSON parser). Do not use `grep`, `head`, or partial line scraping as the source of truth for API decisions.
- On list/read endpoints, inspect one real payload item and use the exact field names returned by that host. Do not invent compatibility names such as `createdAt` or `updatedAt` when the live payload uses fields like `creationDate` or `modificationDate`.
- For `GET /api/scripts/{name}/charts/data` and `GET /api/agents/{name}/charts/data`, keep `maxBars` within `1..10000`. If you need more history, page with `skipBars` while keeping each request at `<=10000`; do not probe `25000`, `50000`, `100000`, or similar values hoping the server will clamp them.
- If you already know an exact built-in template block type name, prefer `GET /api/template-blocks/{typeName}` over downloading the full catalog. Use `GET /api/handlers/{typeName}` for real handler schemas, and if that handler lookup says "not found", check `GET /api/template-blocks/{typeName}` before concluding the type is unavailable.
- Treat `GET /api/template-blocks` as the catalog for template items such as sources, open/close/change position blocks, cycle blocks, ValueUpdater, panes, and similar graph primitives. Indicator, transform, compression, and calculation families are often handlers instead. If a filtered `template-blocks?query=...` call returns empty, switch immediately to `GET /api/handlers?query=...&top=20`, `GET /api/handlers/search?q=...&limit=20`, `GET /api/handlers/{typeName}`, or `GET /api/handlers/{typeName}/recipe` instead of repeating broader `template-blocks` queries.
- If the missing family name starts with `OpenPosition`, `ClosePosition`, `ChangePosition`, `Value`, or contains `CustomHandlerItem` / `ValueUpdater`, treat it as template-block-first discovery. After one empty `GET /api/handlers/search?...` for such a family, switch immediately to `GET /api/template-blocks?query=...&top=20` and then continue to `/ops` instead of repeating handler discovery.
- Do not treat arbitrary scratch files in the workspace root or `./tmp/` as trusted examples or source-of-truth inputs. Only use local files that the prompt explicitly named or that you created yourself in the current turn.
- For exact-artifact create/repair tasks, do not open or grep previous-run scratch under `./tmp/`, `./agent-matrix-runs/`, or files such as `ops*.json`, `repair*.json`, `*.prompt.txt`, `*.jsonl`, `*.stdout.log`, or `*.stderr.log` as examples for the current artifact. Those are history, not source-of-truth inputs.
- Do not use OS or container temp roots such as `/tmp`, `/workspace/tmp`, `/var/tmp`, `%TEMP%`, `%TMP%`, or parent directories for packaged-run scratch files. Stay inside the current workspace under `./tmp/<folder>` only; if a tool rejects an external temp path, switch immediately to a workspace-local temp path instead of retrying the same external location.
- If you need a local request-body file, prefer stable current-workspace filenames such as `./ops.json`, `./tmp_repair_ops.json`, or one fresh file under `./tmp/<folder>/` when you truly need multiple separate bodies. Do not reuse stale older-run scratch JSON/XML files as examples or as-is.
- After a local write or edit tool reports that a request-body file was created or updated successfully, reuse that exact same relative path immediately in the next POST step. Do not spend another step on `glob`, `rg`, or path-verification hunts. Only troubleshoot the path if the POST itself says `Body file not found`.
- For short single-artifact authoring tasks, do not spend turns on repo-wide grep, broad workspace scans, or deep reference chasing once one or two targeted `template-blocks` / `handlers` reads already gave the missing contract. After that, the next step should be a mutation or lifecycle check on the same artifact.
- When a `curl` URL contains query parameters such as `?query=...&limit=20`, quote the whole URL so shells do not split on `&`.
- For JSON mutation bodies in PowerShell, prefer `Invoke-RestMethod` or a UTF-8 file with `curl --data-binary`; avoid inline `curl -d "{...}"` because quoting can corrupt the body.
- For `POST /api/scripts/{name}/ops`, `/repair/*`, and `intent-check` in PowerShell, prefer `./api-json.ps1 POST <url> <jsonFile>` over raw `Invoke-RestMethod` when you need contract feedback. The packaged helper preserves the JSON error body on 4xx responses, while bare `Invoke-RestMethod` often collapses it into an opaque exception and causes blind retry loops.
- The same PowerShell rule applies to create and template-create routes: if the command needs a JSON body, do not nest it inside `powershell -Command`. Use the current shell plus `./api-json.ps1`, or write the body to `./tmp/*.json` and post it directly from the current shell.
- For body-less localhost reads such as `GET /api/status`, `/health`, `/authoring-quality`, `/ui`, `/explain`, or `/metrics-summary`, use the packaged helper `./api-json.ps1 GET /api/...` with no body file or use `Invoke-RestMethod`. The same packaged helpers also support body-less POST routes such as `POST /api/scripts/{name}/repair/authoring-quality`; call them as `./api-json.ps1 POST /api/...` or `./api-json.sh POST /api/...` with no body file. For PowerShell JSON-body mutation routes such as `POST /api/scripts/{name}/instrument-source`, `POST /api/scripts/{name}/intent-check` with a request body, `POST /api/scripts/{name}/ops`, `POST /api/scripts/{name}/visualize`, and `POST /api/scripts/{name}/parameterize`, default to the packaged helper `./api-json.ps1 POST /api/... <jsonFile>` or to `Invoke-RestMethod`.
- Canonical helper examples should stay in `/api/...` form, but the packaged `api-json` helpers also normalize localhost-relative `scripts/...` and `/scripts/...` routes back under `/api/` when a model drops that prefix. Treat that as a safety net, not as the preferred style.
- The packaged workspace ships shell-matching JSON write wrappers for localhost API routes:
  - PowerShell body-less read: `./api-json.ps1 GET /api/<route>`
  - PowerShell body-less POST: `./api-json.ps1 POST /api/<route>`
  - PowerShell file body: `./api-json.ps1 POST /api/<route> ./ops.json`
  - PowerShell stdin body: `@'{"request":{...}}'@ | ./api-json.ps1 POST /api/<route> -`
  - Bash/POSIX body-less read: `./api-json.sh GET /api/<route>`
  - Bash/POSIX body-less POST: `./api-json.sh POST /api/<route>`
  - Bash/POSIX file body: `./api-json.sh POST /api/<route> ./ops.json`
  - Bash/POSIX stdin body: `cat body.json | ./api-json.sh POST /api/<route> -`
- The packaged workspace also ships direct create helpers for `POST /api/script-manager/scripts`:
  - PowerShell: `./create-script.ps1 "<leafName>" ["<parentPath>"]`
  - Bash/POSIX: `./create-script.sh "<leafName>" ["<parentPath>"]`
- Prefer those direct create helpers for exact new-script creation instead of hand-escaping the create JSON body.
- For `./create-script.ps1` or `./create-script.sh`, the canonical form is the leaf script name plus the parent folder path as separate arguments, for example `./create-script.ps1 "My Script" "AI/KaufmanStrategies"`. If `parentPath` is omitted, the packaged helper now also tolerates one foldered full name and auto-splits the last path segment as a forgiving fallback, but do not rely on that fallback when you can pass separate arguments explicitly.
- On Windows PowerShell, prefer `./create-script.ps1` or `./api-json.ps1 POST /api/script-manager/scripts <bodyFile>` over raw `Invoke-RestMethod` for the blank-create step. A wrong-route `POST /api/scripts` 4xx often collapses into an opaque exception there and hides the server's route-correction body.
- If a name-only `POST /api/scripts` returns HTTP 400 on an ordinary blank-create workflow, treat that as proof you used the code-first route by mistake. Switch directly to `POST /api/script-manager/scripts` or `./create-script.ps1`; do not burn the next step on `/api/scripts` list reads just to diagnose a name collision.
- Prefer those packaged wrappers over inline `curl -d "{...}"` when you need a JSON body for `create`, `/ops`, `/repair`, `/parameterize`, `/visualize`, and similar localhost mutation routes.
- If `/ops` returns `InvalidOp`, the server already parsed the JSON and the problem is the `ops[]` entry shape. Fix the body to use explicit `op` fields plus canonical keys; do not keep changing shell quoting on the same malformed payload.
- For `ConnectByInputName`, `toInputName` must be a real destination input name from the host contract, not an output-port name such as `Out`. If the destination input name is blank in `inputs[]`, treat that input as positional: use the host's real positional token from `/explain` or switch to a positional connect form instead of guessing a placeholder name.
- If `/explain` shows unnamed or duplicate compatible inputs on a multi-input block such as `TwoOrMoreInputsItem`, do not invent `Input1` / `Input2` placeholders and do not use `ConnectAuto`. Use `Connect` with the exact zero-based `toPort` index for each input.
- For `AddPane`, either omit `position` or pass the documented structured position object. Do not send a scalar numeric `position`.
- `paneKey` values created by `AddPane` are UI pane identifiers, not graph block IDs. Do not target a pane with `Connect`, `ConnectByInputName`, or ordinary block-input wiring. Use `AddGraphLink` for visible chart series and keep block wiring inside graph blocks only.
- The inverse also applies: do not target graph blocks with `AddGraphLink` unless the destination is a real pane side such as `RIGHT` or `LEFT`. Non-pane endpoints belong to ordinary block wiring.
- On an unfinished authoring task, do not stop to emit a progress summary, a plan-only handoff, or a follow-up question such as "how would you like me to begin?" while the same artifact still has a concrete localhost repair path. After a failed `/ops` batch, use the returned exact `errors[]`, one exact contract read if needed, and then retry the same artifact instead of asking the user what to do next.
- For `POST /api/script-manager/scripts`, use the documented create request object rather than inventing `scriptType` or ad hoc top-level fields. The safe generic body shape is:
  - `{ "request": { "name": "<leafName>", "type": "Script", "parentPath": "" } }`
- If you use `curl.exe --data-binary` with a JSON file, reference the file contents exactly as `--data-binary "@ops.json"`; do not pass plain path strings like `--data-binary "C:\\...\\ops.json"` or `--data-binary "./ops.json"`.
- If the current shell is Bash/POSIX, prefer workspace-relative file bodies such as `--data-binary @./tmp/ops.json` or `--data-binary @tmp/ops.json` with forward slashes. Avoid absolute `@C:\...` file references and PowerShell-style quoting in Bash.
- If you send a JSON body from a file with `curl`, use exactly one request-body flag, for example `curl --data-binary "@ops.json" ...`. Do not combine `-d @ops.json` with `--data-binary` in the same command.
- In PowerShell, do not use bare `curl -d @file.json` tokens for localhost JSON routes. Use the packaged wrapper or `curl.exe --data-binary "@./tmp/file.json"` instead.
- For `/ops` and other JSON mutation routes, the request body must stay a JSON object, not a pre-escaped JSON string. For `/ops`, send `{ "ops": [ ... ] }`, never a bare array.
- When you `AddBlock` and plan to reference that block in later ops, assign a stable `blockId` or `codeName` yourself in the `AddBlock` op. Do not rely on auto-generated names like `DoubleCustomHandlerItem1`.
- Do not use `ConnectAuto` in the first real trade-path `/ops` batch for `BoolCustomHandlerItem`, `DoubleCustomHandlerItem`, `OpenPosition*`, `ClosePosition*`, `ChangePosition*`, `ClosePositionByStopItem`, `ClosePositionByProfitItem`, `Decompress`, `Decompres`, or `DecompressBool`. Those families often have multiple compatible inputs or role-sensitive ports; read the exact input names and use `ConnectByInputName` or `Connect`.
- If `/ops` returns `AmbiguousAutoConnect` or `NoCompatibleInput`, treat that as a blocking contract error. Read exact `inputs[]` from `/explain` plus one exact `template-blocks` or `handlers` contract if needed, then resend a corrected `/ops` batch on the same artifact immediately. Do not continue lifecycle or broad discovery first.
- For ordinary runtime handlers, do not invent literal generic carrier types such as `Handler` in `AddBlock`. Either omit `blockType` and let the server resolve the canonical carrier, or use the exact container/template type that current localhost metadata already proved for that family.
- If `GET /api/handlers/search?...`, `GET /api/handlers/{typeName}`, or `GET /api/handlers/{typeName}/recipe` already exposes `preferredBlockType` or `canonicalAddBlockOp.blockType`, reuse that exact carrier in `AddBlock` instead of inventing `Handler` or guessing from memory.
- For ordinary source/indicator runtime handlers such as `Close`, `Open`, `High`, `Low`, `Bollinger*`, `Highest`, `Lowest`, `StDev`, `EMA`, and similar handler families, do not persist the raw family name as `blockType`. Use the server's `canonicalAddBlockOp` or omit `blockType` so `/ops` resolves the required handler container such as `ConverterItem`, `TwoOrMoreInputsItem`, or another advertised carrier.
- If formulas or downstream blocks report missing inputs even though visible links target such a price/indicator handler, inspect `/explain`; if the upstream block persisted as raw `TypeName=Close` / `Open` / similar instead of a handler container with `HandlerTypeName`, repair that block shape on the same artifact before rewriting formulas.
- Canonical `/ops` mutation names include `AddBlock`, `Connect`, `ConnectAuto`, `ConnectByInputName`, `SetSourceMapping`, `AddPane`, `AddGraphLink`, `AddControlPane`, `AddControlLink`, `UpdateControlPane`, `SetOptimizationRange`, and `SetBlockLocation`. There is no generic `Add`, `AddSource`, or `AddGraphPane` op.
- If you write JSON to a file for `curl --data-binary`, write UTF-8 without BOM when possible. On Windows PowerShell, `Invoke-RestMethod` is usually safer than hand-escaped inline JSON for `/ops`.
- On Windows PowerShell, do not send single-quoted inline JSON to `curl.exe -d` for `/ops`; prefer `Invoke-RestMethod` or a UTF-8 file with `curl.exe --data-binary`.
- For graph ops, use canonical field names only: `blockId`, `typeName`, `blockType`, `category`, `fromBlockId`, `fromPort`, `toBlockId`, `toPort`, `toInputName`. Do not invent aliases such as `handlerType`, `handlerTypeName`, `blockKey`, `fromBlock`, `toBlock`, or `toPortNum` inside `/ops`.
- If a higher-timeframe branch must remain editable or optimizable, prefer `CompressAdvanced` with a fixed `IntervalBase` and expose or optimize only `Interval`. If several compressed branches must share the same higher-timeframe parameter, use `ParameterShareItem` or `OneToManyParametersShareItem` rather than duplicating inconsistent constants.
- If you calculate indicators or signals on a compressed security and then use them back on the base trade path, ensure they return effectively into that base trade/filter path. An explicit `Decompress` / `DecompressBool` block linked to the same `Compress` block is one common pattern when the host requires it. Do not rely on orphaned compressed side branches as proof that the base trade path is using higher-timeframe data.
- For upstream price-series blocks such as `Close`, `Open`, `High`, or `Low` on either the base or compressed branch, verify immediately after creation that the required `SECURITY` input is connected and the block resolved correctly in `/explain`, `validate`, or `build`. Do not continue wiring formulas, exits, or comparisons off a price-series block that still has a missing input or already fails `validate` or `build` with a missing-input error.
- When the host requires an explicit return block, use `Decompress` / `DecompressBool` exactly once per compressed branch that must return to the base trade path. After a numeric or boolean series is already back on the base timeframe, compare or combine it there directly; do not pass that already-aligned base-timeframe series through another `Decompress*` block.
- `Decompress` / `DecompressBool` are for restoring compressed-branch outputs to the base source length. Their `SECURITY` input should come from the exact upstream `Compress` / `CompressAdvanced` block that produced the branch, and the second input should be the still-compressed numeric or boolean stream from that same branch. Do not feed them the base `Src`, and do not feed them a signal that was already computed on the base timeframe or already aligned to the base source.
- `Decompress` / `Decompres` outputs are numeric or boolean base-timeframe series, not tradable security sources. Use decompressed outputs for formulas, indicator inputs, `Eq`, `Prc`, thresholds, or plotted overlays, but never as `OpenPosition*.Src` or `RelativeCommission.SECURITYSource`.
- If a higher-timeframe branch must influence entry condition, order price, stop price, or indicator visualization, keep the tradable security path on the base `Src` and ensure the higher-timeframe output returns effectively into the base-timeframe formula/Eq/Prc/overlay path. A common explicit shape is `Compress* -> compressed price/indicator/condition -> Decompress* -> base-timeframe formula/Eq/Prc/overlay`, but a host-supported equivalent return path can also satisfy that requirement.
- Once `GET /api/handlers/{typeName}` or `/explain` already shows that `Decompress` / `Decompres` returns `DOUBLE` or `BOOL`, do not stop on a `SECURITY` vs `DOUBLE` explanation and do not ask the user what to do next. That is the normal contract. Continue by wiring the decompressed output into a base-timeframe formula, `Eq`, `Prc`, threshold, overlay, or separate-pane series while the tradable security path stays on the base `Src`.
- For a higher-timeframe filter or confirmation prompt, the first valid base-path consumer of the decompressed branch is usually a base-timeframe `Eq` gate, comparison/formula branch, stop/profit `Prc`, or a plotted overlay/separate-pane series. Do not search for a special trade block that accepts `AnyOption` or another non-default carrier just to force a decompressed `DOUBLE` into trade `Src`.
- On a strategy prompt, do not create a second compressed helper branch, duplicate compressed RSI branch, or another explicit `Compress* -> Decompress*` pair before the first compressed branch already feeds one real base-path consumer on the same artifact. One effective higher-timeframe branch plus one real base-path consumer is enough for the first proof; duplicate branches before that are usually orphaned noise.
- Once these docs plus `GET /api/handlers/{typeName}` already define the `Compress*` / `Decompress*` contract for the current artifact, stop hunting local examples. Do not grep the workspace or `./tmp/` for older `Decompress*` snippets, request bodies, or repair logs.
- If a tool, helper, or shell step asks for an external temp or repo-level scratch path such as `C:\Projects\dev30\tmp\...`, stop that side path and recreate the request body under `./tmp/` inside the current packaged workspace instead of retrying the external location.
- If a template discovery response includes example commands, reuse the advertised route/body directly. PowerShell examples can be used verbatim only when the current shell is PowerShell; otherwise translate the same route/body into the current shell's native client.
- Use `curl --data-binary "@file.json"` only for large UTF-8 JSON bodies. On Windows PowerShell, call `curl.exe` to avoid the `curl` alias.
- If a localhost API probe fails because `curl` resolved to a shell alias, the shell starts in an unexpected root directory, or a non-API tool asks for unrelated filesystem permissions, stop debugging that side path and switch immediately to direct `curl.exe` or `Invoke-RestMethod` calls against the localhost route.
- If the current workspace already contains the local docs you need, do not glob or read parent/source-tree duplicates of those same docs. If a tool rejects an outside-workspace read or `external_directory` path, stop that side path immediately and continue with the current workspace docs plus localhost routes only.
- Normalize provider casing against `GET /api/datasources` before deciding that a named datasource is missing. In `/api/datasources` list responses, use `name` as the datasource identifier; `dataSourceName` may be empty or legacy in some projections. Use `GET /api/agent-manager/securities` only with a concrete non-empty instrument query and the canonical shape `GET /api/agent-manager/securities?dataSourceName=...&query=...&limit=20`; do not use it for broad enumeration.
- For multipart file uploads such as `/api/indicator-dlls`, prefer `curl -F` with the DLL you just built from the current temp workspace, for example `curl -F "file=@<path-to-built-dll>" ...`.   - when that script needs a tradable instrument source, use `POST /api/scripts/{name}/instrument-source`, not `/source`
  - if the prompt already names the instrument, reuse it directly as `securityId`; only if the exact id is still unknown, query `GET /api/agent-manager/securities?dataSourceName=...&query=...&limit=20`
  - requested datasource, security, and base timeframe are task constraints; during performance repair reduce history length, active `UseDateFrom`/`DateFrom` plus `UseDateTo`/`DateTo` proof window, or indicator defaults instead of changing the prompt timeframe to a faster interval
  - for offline/cached visual indicator or analysis proof, stay on script lifecycle/proof routes and do not create or mutate `agent-manager/agents` unless the user explicitly asked for a runtime agent
- Stop if failed:
  - do not fall back to repo `scripts/*.cs`
  - do not use `ExternalScriptItem` unless the user explicitly provided a server-side file path
  - do not assume `Invoke-RestMethod -Form` exists or behaves the same on every shell; on Windows PowerShell 5.1 use `curl.exe -F`
  - do not browse a third or fourth source

## Phase 1: Create

Use for a brand-new script.

- First call for a supported template-backed strategy:
  - `GET /api/scripts/templates`
  - then `POST /api/scripts/templates/{templateName}` only if a listed template clearly matches the requested workflow
- if a listed template returns `nextStepHint`, `exampleCreatePowerShell`, `exampleApplyPowerShell`, or example body JSON, use that advertised `POST` route next instead of repeating discovery
- if the prompt also requires an exact script name or exact folder placement, prefer the advertised `applyRoute` or `applyByNameRoute` on that exact artifact instead of a template create route that mints a temporary name
- For a multi-indicator chart/dashboard where all indicators must share one editable and optimizable Period parameter:
  - for generic multi-indicator chart/dashboard authoring, prefer blank create plus explicit `/ops` over disabled helper panel templates
  - body shape:
    - `{ "name": "<scriptName>", "dataSourceName": "<dataSourceName>", "securityId": "<securityId>", "interval": "<interval>", "indicators": ["<indicatorA>", "<indicatorB>", "<indicatorC>"], "sharedParameterValue": 20 }`
- First call for a generic blank script:
  - `POST /api/script-manager/scripts`
- Fields to read:
  - `response.scriptName`
  - `response.scriptId`
  - for template-created scripts, also keep `response.defaultOptimizationRequest` when present
  - fallback: `response.name`, `response.id`, then `response.data`
- Next allowed call:
  - `POST /api/scripts/{name}/validate`
- After template discovery, do not repeat discovery calls in the same phase. Move directly to create/apply.
- After one successful filtered family lookup such as `GET /api/template-blocks?query=...`, `GET /api/handlers?query=...`, or the exact `/{typeName}` contract, do not repeat that same query string again in the same phase unless the latest mutation or live error proved the family itself is still unresolved. Use the current artifact, `/ops`, `/explain`, or the next lifecycle blocker instead.
- On a brand-new blank script, do not call `/explain` before the first `/ops` batch.
- On a brand-new blank script, do not spend extra steps on `/explain`, broad diagnostics, or shell quoting experiments before the first real `/ops` batch.
- Stop if failed:
  - do not continue to `/build` or `/run`

## Phase 2: Run

Use after a successful create or when an existing exact script name is already known.

- First call:
  - `POST /api/scripts/{name}/validate`
- Then:
  - `POST /api/scripts/{name}/build`
  - `POST /api/scripts/{name}/load`
  - `POST /api/scripts/{name}/run`
- Final read for this phase:
  - `GET /api/scripts/{name}/metrics-summary`
- Fields to read:
  - `response.data.netProfit`
  - `response.data.allTrades`
  - `response.data.profitFactor`
  - `response.data.maxDrawdownPct`
- Stop if failed:
  - do not switch to guessed `/graph`, `/edit`, or repo files

## Phase 3: Optimization start

Use only for `same script` / `this script` optimization.

- First call:
  - if the user explicitly limited the optimization to certain parameter families or gave a run cap, first inspect `GET /api/scripts/{name}/parameters` and keep only the approved families in the job-scoped selected subset before `optimization/start`; use `selectedParameterIds` when the requested subset is narrower than the persisted defaults
  - for foldered scripts, use the folder-safe route shape that the helpers generate: `POST /api/scripts/optimization/start/by-name/{**name}`, then `GET /api/scripts/optimizations/{jobId}`, `GET /api/scripts/optimizations/{jobId}/best/by-name/{**name}`, and `POST /api/scripts/optimizations/{jobId}/apply/by-name/{**name}`; do not inline a slash-containing script name into `/api/scripts/{name}/...`
  - if the create/template response includes `defaultOptimizationRequest`, use it directly:
    - `POST /api/scripts/{name}/optimization/start`
  - otherwise:
    - `GET /api/scripts/{name}/lab-options`
    - post that full `response.data.options` back with `rtUpdates=false`
    - `POST /api/scripts/{name}/optimization/start`
- Fields to read:
  - `response.jobId`
  - `response.scriptName`
  - `response.status`
  - `response.data.optimizationParameterCount`
  - `response.data.searchSpace`
  - `response.data.plannedIterations`
  - `response.data.trackedMetrics`
- Next allowed call:
  - `GET /api/scripts/optimizations/{jobId}`
- Stop if failed:
  - do not create/copy/template a new script

## Phase 4: Best and apply

Use only after the optimization job is complete.

- First call:
  - `GET /api/scripts/{name}/optimizations/{jobId}/best?metric=NetProfit&rank=1`
- Fields to read:
  - `response.jobId`
  - `response.rank`
  - `response.parameterValues`
  - `response.suggestedValue`
  - `response.data.netProfit`
  - `response.data.profitFactor`
  - `response.data.maxDrawdownPct`
  - `response.data.allTrades`
- Next allowed call:
  - `POST /api/scripts/{name}/optimizations/{jobId}/apply?metric=NetProfit&rank=1`
- Then:
  - `POST /api/scripts/{name}/build`
  - `POST /api/scripts/{name}/run`
  - `GET /api/scripts/{name}/metrics-summary`
  - `GET /api/scripts/{name}/messages`
  - `GET /api/scripts/{name}/logs`
- Stop if failed:
  - do not infer the winner from `runId` or a formatted table
  - do not treat `rank=1` as a promoted candidate by itself; if the best row has `allTrades=0`, negative `NetProfit`, `profitFactor < 1`, absurd drawdown, or fails the task's explicit trade-count/PF/DD guardrails, treat that as evidence the hypothesis did not clear the bar
  - when trade density or robustness matters, inspect at least one orthogonal Top-K slice such as `AllTrades` or `ProfitFactor`, or another denser positive row, before finalizing
